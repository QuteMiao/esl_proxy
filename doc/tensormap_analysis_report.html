系统概述

TensorMap 是一个轻量级、无依赖的生产者查找映射（Producer-Lookup Map），专门设计用于在 DAG 执行引擎中检测任务间的数据依赖关系。该系统通过维护一个内存中的映射表，记录每个内存区域的生产者任务信息，从而在消费者任务提交时能够自动发现其数据依赖关系。

核心设计理念

TensorMap 采用"零分配"设计理念：所有状态都保存在调用者提供的原始缓冲区中，通过偏移量访问而非指针，使得映射表可以自由移动（memcpy）而无需指针修复。这使得它成为一个完全无耦合的可选工具，可以叠加在任何现有的 DAG 执行引擎之上。

核心特性

  • 生产者查找：给定一个地址区域，快速找到所有产生该数据的任务
  • 子区域重叠检测：支持 2D 区域的几何重叠判断，而不仅仅是精确地址匹配
  • 延迟失效：使用每环（per-ring）水位线机制实现高效的过期条目清理
  • 池化条目：预分配的条目池，避免运行时内存分配
  • 位置无关：纯偏移量设计，支持跨进程/跨设备传输
  • 双模式支持：支持完整 Tensor 几何信息模式和简单缓冲区模式

核心数据结构

2.1 配置结构 (TmConfig)

配置结构定义了 TensorMap 的基本参数，所有参数必须是 2 的幂次方（除了 num_rings）：

/* Configuration POD. num_buckets and every task_window[r] must be powers of two. */ typedef struct TmConfig { uint32_t num_buckets; // 哈希桶数量（必须是2的幂） uint32_t pool_size; // 条目池大小 uint32_t num_rings; // 环层数量 uint32_t task_window[TM_MAX_RINGS]; // 每个环的任务窗口大小 } TmConfig;

2.2 区域描述符 (TmRegion)

描述一个 2D 内存区域的精确几何信息，用于重叠检测：

/* Region descriptor for unit tests (2D row-major geometry). */ typedef struct TmRegion { uint64_t base_addr; // 基础地址 uint32_t storage_numel; // 存储总元素数 uint16_t elem_size; // 元素大小（字节） int32_t version; // 版本号（用于懒失效） uint16_t strides[TM_MAX_DIMS]; // 步长数组 uint16_t shapes[TM_MAX_DIMS]; // 形状数组 uint16_t offsets[TM_MAX_DIMS]; // 偏移数组 } TmRegion;

2.3 池条目 (TmEntry)

映射表中的核心条目，存储生产者的完整信息：

/* Pool entry. Links are pool indices (-1 = none) so the buffer is relocatable. */ typedef struct TmEntry { uint64_t base_addr; // 基础地址 uint64_t producer_id; // 生产者ID（环+本地编码） int32_t version; // 版本号 uint16_t elem_size; // 元素大小 uint16_t strides[TM_MAX_DIMS]; // 步长 uint16_t shapes[TM_MAX_DIMS]; // 形状 uint16_t offsets[TM_MAX_DIMS]; // 偏移 int32_t next_in_bucket; // 桶内下一条目索引 int32_t prev_in_bucket; // 桶内前一条目索引 int32_t next_in_task; // 任务链下一条目索引 int32_t prev_in_task; // 任务链前一条目索引 int32_t bucket_index; // 所属桶索引（-1表示未链接） } TmEntry;

哈希链结构

每个 TmEntry 通过 next_in_bucket 和 prev_in_bucket 形成哈希桶链表，实现 O(1) 平均查找。

任务链结构

每个 TmEntry 通过 next_in_task 和 prev_in_task 形成任务链表，用于批量失效清理。

2.4 头部结构 (TmHeader)

管理整个映射表元数据：

/* In-buffer header: config echo + cursors + sub-region offsets. */ typedef struct TmHeader { TmConfig cfg; // 配置副本 int32_t next_entry_idx; // 下一个可用条目索引 int32_t free_num; // 空闲条目数量 int32_t last_alive[TM_MAX_RINGS]; // 每环最后活跃任务ID int32_t last_cleanup[TM_MAX_RINGS]; // 每环最后清理任务ID uint32_t off_buckets; // 哈希桶数组偏移 uint32_t off_pool; // 条目池偏移 uint32_t off_free; // 空闲列表偏移 uint32_t off_task_heads[TM_MAX_RINGS]; // 每环任务头数组偏移 } TmHeader;

内存布局

TensorMap 采用单一原始缓冲区，所有数据结构紧凑排列，使用偏移量进行寻址：

📦 单一内存缓冲区布局
Offset 0: TmHeader
配置 + 游标 + 子区域偏移
↓
off_buckets: int32_t[]
哈希桶数组（num_buckets 个）
↓
off_pool: TmEntry[]
条目池（pool_size 个条目）
↓
off_free: int32_t[]
空闲列表（pool_size 个索引）
↓
off_task_heads[ring]: int32_t[]
每环任务头数组（task_window[ring] 个）

内存计算公式

所需字节数 = sizeof(TmHeader) + num_buckets × sizeof(int32_t) + pool_size × (sizeof(TmEntry) + sizeof(int32_t)) + Σ(ring) task_window[ring] × sizeof(int32_t)

默认配置：2048 桶 + 8192 条目池 + 65536 任务窗口 ≈ 500KB

内存布局计算代码

static inline uint64_t tm_layout(const TmConfig *cfg, TmHeader *out) { uint64_t cur = sizeof(TmHeader); uint64_t off_buckets = cur; cur += (uint64_t)cfg->num_buckets * sizeof(int32_t); uint64_t off_pool = cur; cur += (uint64_t)cfg->pool_size * sizeof(TmEntry); uint64_t off_free = cur; cur += (uint64_t)cfg->pool_size * sizeof(int32_t); uint64_t off_task[TM_MAX_RINGS] = {0}; for (uint32_t r = 0; r < cfg->num_rings; r++) { off_task[r] = cur; cur += (uint64_t)cfg->task_window[r] * sizeof(int32_t); } if (out != NULL) { out->off_buckets = off_buckets; out->off_pool = off_pool; out->off_free = off_free; for (uint32_t r = 0; r < cfg->num_rings; r++) { out->off_task_heads[r] = off_task[r]; } } return cur; }

生产者ID编码

TensorMap 使用 64 位整数编码生产者身份，自包含环和本地 ID 的编码逻辑：

/* Producer-id encoding (owned by this module). */ static inline uint64_t tm_make_id(uint32_t ring, uint32_t local) { return ((uint64_t)ring << 32) | local; } static inline uint32_t tm_ring_of(uint64_t id) { return (uint32_t)(id >> 32); } static inline uint32_t tm_local_of(uint64_t id) { return (uint32_t)(id & 0xFFFFFFFFu); }
Ring ID (32bit)
+
Local ID (32bit)
=
Producer ID (64bit)

编码规则

高 32 位存储环 ID，低 32 位存储本地任务 ID。这种编码方式允许单次比较同时获取两个维度。

解码操作

tm_ring_of() 和 tm_local_of() 提供常量时间复杂度的解码，支持快速的任务链管理和失效判断。

数据依赖检测算法

3.1 区域重叠检测 (Overlap Detection)

重叠检测是 TensorMap 的核心算法，用于判断两个 2D 区域是否存在数据依赖：

static inline TmOverlap tm_overlap(const TmRegion *in, const TmEntry *e) { // 版本号检查：输入版本必须不大于条目版本 if (in->version > e->version) return TM_OVERLAP_OTHER; // 元素大小检查：必须匹配 if (in->elem_size != e->elem_size) return TM_OVERLAP_OTHER; // 步长检查：所有维度步长必须一致 for (uint16_t i = 0; i < TM_MAX_DIMS; i++) { if (in->strides[i] != e->strides[i]) return TM_OVERLAP_OTHER; } // 2D 超矩形重叠检测 bool covered = true; for (uint16_t i = 0; i < TM_MAX_DIMS; i++) { const uint64_t a0 = in->offsets[i], a1 = a0 + in->shapes[i]; const uint64_t b0 = e->offsets[i], b1 = b0 + e->shapes[i]; // 无重叠检测：A1 <= B0 或 B1 <= A0 if (!(a1 > b0 && b1 > a0)) return TM_OVERLAP_NONE; // 完全覆盖检测：A0 <= B0 且 B1 <= A1 if (!(a0 <= b0 && b1 <= a1)) covered = false; } return covered ? TM_OVERLAP_COVERED : TM_OVERLAP_OTHER; }

3.2 重叠状态定义

TM_OVERLAP_NONE (0)

两个区域完全不相交，无数据依赖关系。

TM_OVERLAP_COVERED (1)

消费者区域被生产者区域完全覆盖（生产者是消费者的超集）。

TM_OVERLAP_OTHER (2)

区域部分重叠（保守判断），可能存在数据依赖。

3.3 2D 重叠检测原理

A区域: [a0, a1]
vs
B区域: [b0, b1]

维度 0 (行):

• 重叠条件: a1 > b0 && b1 > a0

• 覆盖条件: a0 <= b0 && b1 <= a1


维度 1 (列):

• 重叠条件: a1 > b0 && b1 > a0

• 覆盖条件: a0 <= b0 && b1 <= a1

3.4 哈希函数

使用 Golden Ratio 哈希实现均匀分布：

static inline uint32_t tm_hash(const TmTensorMap *self, uint64_t key) { key *= 0x9E3779B97F4A7C15ULL; // 乘以 Golden Ratio 的倒数 return (uint32_t)(key >> (64 - __builtin_ctz(tm_hdr(self)->cfg.num_buckets))); }
注意：哈希函数要求 num_buckets 必须是 2 的幂次方，以便使用位移操作代替取模，提高性能。

依赖检测流程

4.1 插入流程 (tm_insert)

当任务产生一个输出张量时，将其注册到 TensorMap：

1. 分配条目
→
2. 填充数据
→
3. 加入哈希桶
→
4. 加入任务链
static inline void tm_link_entry(TmTensorMap *self, int32_t idx, uint64_t addr, uint64_t producer_id) { TmEntry *pl = tm_pool(self); int32_t *bk = tm_buckets(self); TmEntry *e = &pl[idx]; e->producer_id = producer_id; // 哈希计算并加入桶链 const uint32_t b = tm_hash(self, addr); e->bucket_index = (int32_t)b; e->prev_in_bucket = -1; e->next_in_bucket = bk[b]; if (bk[b] != -1) pl[bk[b]].prev_in_bucket = idx; bk[b] = idx; // 加入任务链（用于失效清理） const uint32_t ring = tm_ring_of(producer_id); const uint32_t slot = tm_local_of(producer_id) & (tm_hdr(self)->cfg.task_window[ring] - 1); int32_t *th = tm_task_heads(self, ring); e->prev_in_task = -1; e->next_in_task = th[slot]; if (th[slot] != -1) pl[th[slot]].prev_in_task = idx; th[slot] = idx; }

4.2 查找流程 (tm_lookup)

当任务声明输入依赖时，查找所有重叠的生产者：

1. 哈希定位桶
→
2. 遍历桶链
→
3. 验证有效性
→
4. 检测重叠
→
5. 回调生产者
static inline void tm_lookup(TmTensorMap *self, const TmRegion *r, TmMatchFn on_match, void *ctx) { // 步骤1：哈希定位 const uint32_t b = tm_hash(self, r->base_addr); int32_t cur = tm_buckets(self)[b]; TmEntry *pl = tm_pool(self); // 步骤2-5：遍历并检测 while (cur != -1) { const int32_t next = pl[cur].next_in_bucket; TmEntry *e = &pl[cur]; // 验证有效性：检查水位线 if (tm_entry_valid(self, e) && e->base_addr == r->base_addr) { const TmOverlap st = tm_overlap(r, e); if (st != TM_OVERLAP_NONE) { if (!on_match(e, st, ctx)) return; } } cur = next; } }

4.3 有效性检查 (tm_entry_valid)

基于水位线机制判断条目是否仍然有效：

static inline bool tm_entry_valid(const TmTensorMap *self, const TmEntry *e) { return (int32_t)tm_local_of(e->producer_id) >= tm_hdr(self)->last_alive[tm_ring_of(e->producer_id)]; }

水位线机制

每个环维护一个 last_alive 水位线，表示该环中"最老的活跃任务ID"。当生产者ID小于水位线时，说明该任务已完成并可以被安全清理。这种机制避免了昂贵的引用计数或显式删除操作。

懒失效与清理机制

5.1 同步机制 (tm_sync)

推进水位线，但不立即清理：

static inline void tm_sync(TmTensorMap *self, uint32_t ring, int32_t last_alive) { tm_hdr(self)->last_alive[ring] = last_alive; }

5.2 批量清理机制 (tm_cleanup_retired)

清理指定范围内的所有过期条目：

static inline void tm_cleanup_retired(TmTensorMap *self, uint32_t ring, int32_t old_alive, int32_t new_alive) { const uint32_t mask = tm_hdr(self)->cfg.task_window[ring] - 1; int32_t *th = tm_task_heads(self, ring); TmEntry *pl = tm_pool(self); // 遍历每个过期任务的任务链 for (int32_t local = old_alive; local < new_alive; local++) { const uint32_t slot = (uint32_t)local & mask; int32_t cur = th[slot]; while (cur != -1) { const int32_t next = pl[cur].next_in_task; tm_free_entry(self, cur); // 归还到空闲池 cur = next; } th[slot] = -1; } tm_hdr(self)->last_cleanup[ring] = new_alive; }

5.3 完整同步流程 (tm_sync_tensormap)

提交任务时的完整同步操作：

static inline void tm_sync_tensormap(TmTensorMap *self, uint32_t ring, int32_t last_alive) { // 1. 推进水位线 tm_sync(self, ring, last_alive); // 2. 清理过期条目（仅当有新过期任务时） const int32_t old = tm_hdr(self)->last_cleanup[ring]; if (last_alive > old) tm_cleanup_retired(self, ring, old, last_alive); }

→
检查清理范围
→

高层依赖接口

6.1 结构 Tensor 模式

使用完整的 Tensor 几何信息进行依赖检测：

static inline void tm_in_ptr(uint16_t tid, const Tensor *t) { add_input_ptr(tid, t); tm_pending_push(t, TM_PEND_IN); } static inline void tm_out_ptr(uint16_t tid, const Tensor *t) { add_output_ptr(tid, t); tm_pending_push(t, TM_PEND_OUT); } static inline void tm_inout_ptr(uint16_t tid, const Tensor *t) { add_inout_ptr(tid, t); tm_pending_push(t, TM_PEND_INOUT); }

6.2 提交流程 (tm_submit)

任务提交时的完整依赖解析逻辑：

static inline void tm_submit(uint16_t tid) { TmCollectCtx ctx = { .consumer = tid, .pn = 0 }; int i; // 阶段1：查找所有输入依赖的生产者 for (i = 0; i < g_tm_pend_n; i++) { if (g_tm_pend[i].kind & TM_PEND_IN) { ctx.is_inout = (g_tm_pend[i].kind == TM_PEND_INOUT); tm_lookup_tensor(&g_tm_map, g_tm_pend[i].t, tm_collect_on_match, &ctx); } } // 阶段2：注册所有输出依赖 for (i = 0; i < ctx.pn; i++) succeed(tid, ctx.preds[i]); for (i = 0; i < g_tm_pend_n; i++) { if (g_tm_pend[i].kind & TM_PEND_OUT) tm_insert_tensor(&g_tm_map, g_tm_pend[i].t, tid); } tm_pending_clear(); // 阶段3：执行任务并同步水位线 submit(tid); tm_sync_tensormap(&g_tm_map, 0, (int32_t)g_min_uncomplete_task); }

阶段 1：依赖收集

遍历所有待处理的输入张量，调用 tm_lookup_tensor 查找重叠的生产者条目，并收集唯一的前驱任务ID。

阶段 2：依赖注册

为收集到的每个前驱任务调用 succeed() 注册依赖关系，并将所有输出张量插入 TensorMap。

阶段 3：任务执行与同步

调用 submit() 提交任务到执行器，然后更新 TensorMap 的水位线以清理已完成任务的条目。

6.3 收集回调 (tm_collect_on_match)

static inline bool tm_collect_on_match(TmEntry *e, TmOverlap st, void *ctx) { TmCollectCtx *c = (TmCollectCtx *)ctx; const uint16_t p = (uint16_t)tm_local_of(e->producer_id); // 去重：检查是否已收集该生产者 if (p != c->consumer) { for (int i = 0; i < c->pn; i++) if (c->preds[i] == p) goto after_pred; if (c->pn < (int)TM_PENDING_MAX_PRED) c->preds[c->pn++] = p; } after_pred: // INOUT 特殊处理：完全覆盖时移除生产者条目 if (c->is_inout && st == TM_OVERLAP_COVERED) tm_remove(&g_tm_map, e); return true; }

双模式对比

TensorMap 支持两种操作模式，编译时选择：

特性 结构 Tensor 模式 简单缓冲区模式
几何信息 完整 2D 形状/步长/偏移 仅基地址
重叠检测 精确子区域检测 仅地址匹配
适用场景 分块矩阵运算 简单缓冲区管理
编译宏 USE_TENSORMAP USE_TENSORMAP + TENSORMAP_WHOLE_BUFFER
内存效率 需存储完整几何信息 最小化条目大小

模式选择建议

• 结构 Tensor 模式：适用于 AI 模型推理中的复杂张量操作，如 Qwen3 解码场景，需要精确的子区域依赖检测

• 简单缓冲区模式：适用于简单的生产者-消费者场景，只需检测完整缓冲区的读写依赖

test_tensormap.c 中的 2D 区域重叠测试展示了真实 AI 工作负载中的场景：

static void test_2d_row_tile_overlap(void) { // Producer: tile 1 writes rows [16, 32). TmRegion prod = region_2d(base, batch, hidden, 16, tile, hidden, 4); tm_insert(&map, &prod, tm_make_id(0, 3)); // Consumer reading same tile overlaps. HitSink hit = collect(&map, region_2d(base, batch, hidden, 16, tile, hidden, 4)); assert(hit.count == 1 && tm_local_of(hit.producers[0]) == 3); // Consumer reading a different tile does not overlap. HitSink miss = collect(&map, region_2d(base, batch, hidden, 0, tile, hidden, 4)); assert(miss.count == 0); // Consumer reading rows [8, 24) partially overlaps tile 1. HitSink part = collect(&map, region_2d(base, batch, hidden, 8, 16, hidden, 4)); assert(part.count == 1 && part.statuses[0] == TM_OVERLAP_OTHER); }

精确匹配

消费者读取 [16, 32) 区域精确匹配生产者，产生 TM_OVERLAP_COVERED 状态。

无重叠

消费者读取 [0, 16) 区域与生产者 [16, 32) 完全不重叠，返回 TM_OVERLAP_NONE。

部分重叠

消费者读取 [8, 24) 区域部分重叠，产生 TM_OVERLAP_OTHER 状态。

共存性演示

测试展示了 TensorMap 作为可选工具与现有 DAG 引擎的共存能力：

// task 10 and task 11 both write X (e.g. two chunks of one buffer) TmRegion rx = whole_buffer_region(X); tm_insert(&map, &rx, tm_make_id(0, 10)); tm_insert(&map, &rx, tm_make_id(0, 11)); // consumer task 20 reads X -> should depend on {10, 11} HitSink cx = collect(&map, whole_buffer_region(X)); assert(cx.count == 2); // 自动发现两个生产者 // tasks 10,11 complete: watermark advances to 12 uint16_t min_uncomplete = 12; tm_sync(&map, 0, (int32_t)min_uncomplete); HitSink cx2 = collect(&map, whole_buffer_region(X)); assert(cx2.count == 0); // 过期的生产者不再被找到

总结与架构优势

核心优势

  • 零分配设计：所有内存预分配，避免运行时 malloc/freed 的开销和碎片化
  • 位置无关：纯偏移量访问，支持跨进程/跨设备传输，无需指针修复
  • 无耦合：独立的头文件，不依赖 Tensor、ring_buf 或 task 结构
  • 延迟失效：水位线机制避免昂贵的引用计数操作
  • 批量清理：任务完成时批量回收过期条目，分摊清理开销
  • 精确几何：支持 2D 子区域重叠检测，适用于复杂 AI 工作负载

设计原则

纯 C 实现哲学

TensorMap 完全使用 C 语言实现，仅依赖 <assert.h>、<stdint.h>、<string.h> 和 <stdbool.h>，编译为 C99+ 或 C++ 均可。这确保了它可以集成到任何现有的 C/C++ 项目中，包括裸金属环境和嵌入式系统。

应用场景

DAG 执行引擎

作为可选的依赖发现层，自动检测任务间的数据依赖关系。
